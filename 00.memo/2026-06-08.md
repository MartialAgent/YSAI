## [2026-06-08 14:24] python_syntax — `*` 모으기/풀기 개념 정리 카드 (1단계+2단계+미니 통합)

### 1. 가장 큰 규칙: 위치가 방향을 정한다

| 위치 | `*` 의미 | 방향 |
|------|---------|------|
| **정의** `def f(*x)` | 인자들을 **모은다** → 튜플 | 인자 → 컨테이너 |
| **호출** `f(*x)` | 컨테이너를 **푼다** → 인자들 | 컨테이너 → 인자 |

같은 별표가 **정반대**로 작동. 이 비대칭이 전부.

### 2. `*args` 는 무조건 튜플 (예외 없음)

```python
def f(*args): print(args)

f()          # ()           ← 빈 튜플
f(1)         # (1,)         ← 꼬리 콤마! 1개여도 튜플
f([1,2,3])   # ([1,2,3],)   ← list 1개를 담은 튜플
f(1, 2, 3)   # (1, 2, 3)
```

**무엇이 들어오든 통은 튜플 하나로 정해져 있음**. 들어온 게 list/dict/str 이라도 모인 결과 자료형은 안 바뀜.

### 3. 호출 시 `*` 유무 = 1개냐 N개냐

```python
data = [10, 20, 30]

f(data)     # 인자 1개 (리스트 통째) → args = ([10,20,30],)  len=1
f(*data)    # 인자 3개 (풀려서)      → args = (10, 20, 30)   len=3
```

`*` 가 호출 측에 있으면 **그 자리에서 컨테이너가 폭발**해서 여러 인자로 흩어짐.

### 4. 튜플 표기 함정

| 표기 | 의미 |
|------|------|
| `()` | 빈 튜플 |
| `(x)` | **그냥 괄호** (튜플 아님!) |
| `(x,)` | **원소 1개 튜플** (꼬리 콤마 필수) |
| `(x, y)` | 원소 2개 튜플 |

`*args` 결과 출력에서 `(...,)` 형태로 콤마가 따라붙으면 "1개짜리 튜플" 신호.

### 5. `print(a, b)` 의 출력 형식

```python
print(3, (1, 2, 3))
# 출력: 3 (1, 2, 3)        ← 두 값을 공백으로 그냥 이어 붙임
```

- 콤마는 인자 구분일 뿐, **출력에 괄호 안 생김**.
- 출력에서 보이는 `(...)` 나 `[...]` 는 **각 값 자체의 표기** (튜플/리스트).

### 6. `type(x).__name__` 짧게

```python
type(42)            # <class 'int'>    ← 클래스 객체 (지저분)
type(42).__name__   # 'int'            ← 이름 문자열만
```

- 점(`.`) = "안의 속성 꺼내기" (등호 없으면 항상 **읽기**).
- `__name__` = 클래스에 미리 박혀있는 이름표.

---

### 흔한 함정 5가지 체크리스트

| 함정 | 올바른 인식 |
|------|------------|
| `*args` 가 list 일 수 있다? | ❌ **항상 튜플** |
| `(x)` 가 튜플? | ❌ 그냥 괄호. 튜플은 `(x,)` |
| `print(a, b)` 출력에 괄호? | ❌ 공백으로만 연결 |
| `f(x)` 와 `f(*x)` 같다? | ❌ **인자 개수가 달라짐** |
| 정의 `*` 와 호출 `*` 같은 동작? | ❌ **정반대** (모으기 ↔ 풀기) |

---

### 한 줄 요약 (외울 것)

> **정의에선 모으고, 호출에선 푼다. `*` 결과는 튜플 한정.**

---

### 오늘 막힌 지점 (재발 방지)

1. **`*args` 를 list `[...]` 로 답함** — 항상 튜플인 걸 다시 박아야 함.
2. **`print(a, b)` 출력을 `(a, b)` 튜플 모양으로 답함** — print 콤마는 출력에 괄호를 만들지 않음.
3. **`f(data)` 와 `f(*data)` 결과를 같게 봄** — `*` 한 글자가 인자 개수를 1 ↔ N 으로 바꿈.
4. **`type(x).__name__` 의 점(.)을 "정하기"로 오해** — 점은 "꺼내기(읽기)" 동작.

## [2026-06-08 14:35] python_syntax — 튜플의 꼬리 콤마 (`(1,)`)

### 핵심: **튜플을 만드는 건 콤마(`,`) 다.** 괄호는 단순 그루핑일 뿐.

```python
(1)     # → int 1            ← 괄호만 있으면 그냥 숫자
(1,)    # → tuple (1,)       ← 콤마가 비로소 튜플로 만듦
1, 2, 3 # → tuple (괄호 없이도!)
1,      # → tuple (1,)
```

괄호는 수식 우선순위 묶을 때 쓰는 그 괄호와 같음. 튜플 표식이 아님.

### 출력에 꼬리 콤마가 찍히는 이유

파이썬 `repr()` 규칙: **"다시 코드로 복붙해도 같은 값으로 살아나야 한다"**.
- `(1)` 로 찍으면 → 복붙 시 int 1 이 됨 → 정보 손실
- `(1,)` 로 찍어야 → 복붙 시 튜플 그대로

→ 1개짜리 튜플은 모호함 방지를 위해 **꼬리 콤마 필수**.

### 원소 수별 표기

| 원소 수 | 만드는 법 | 출력 |
|---------|----------|------|
| 0개 | `()` (특수) 또는 `tuple()` | `()` |
| 1개 | `1,` 또는 `(1,)` | `(1,)` ← 꼬리 콤마 |
| 2개+ | `1, 2` 또는 `(1, 2)` | `(1, 2)` |

### 비유

콤마는 **"이건 컬렉션이다"** 라는 도장.
괄호는 종이 위에 그린 **테두리 장식**.
- `(1)` = 도장 없는 종이 (그냥 숫자)
- `(1,)` = 도장 한 번 찍힘 (1개짜리 컬렉션)

### 실전에서 자주 보는 패턴

```python
return value,        # 단일 값을 튜플로 감싸 반환
a, = some_iter       # 좌변 콤마로 "1개짜리 언패킹"
x = (1)              # ✗ int (튜플 아님 — 흔한 함정)
x = (1,)             # ✓ tuple
```

### 한 줄 요약

> **튜플은 콤마가 만든다. 괄호는 거들 뿐. 1개짜리 튜플 `(1,)` 의 꼬리 콤마는 "나 튜플이오" 라는 외침.**

## [2026-06-08 16:05] python_syntax — 4단계 컨테이너 합치기 (`[*a, *b]`, `{**d1, **d2}`)

### 핵심: 별표는 함수 호출뿐 아니라 **컨테이너 리터럴 안**에서도 풀린다.

| 풀기 위치 | 예시 | 결과 |
|----------|------|------|
| 함수 호출 시 | `f(*a, **d)` | 인자들로 흩어짐 |
| **컨테이너 리터럴 안** | `[*a, *b]`, `{**d1, **d2}` | 원소로 흩어짐 |

### 결과 자료형은 **바깥 괄호**가 결정

```python
[*a, *b]      # list
(*a, *b)      # tuple
{*a, *b}      # set (중복 제거)
{**d1, **d2}  # dict (** 필수)
```

### dict 합치기의 두 규칙 (가장 헷갈리는 자리)

> **1. 첫 등장이 자리(순서)를 정한다 — 키 위치는 안 바뀜**
> **2. 마지막 쓰기가 값을 정한다 — 같은 키 값은 뒤가 이김**

```python
d1 = {'a': 1, 'b': 2}
d2 = {'b': 99, 'c': 3}

{**d1, **d2}   # → {'a': 1, 'b': 99, 'c': 3}   ← 'a','b','c' 순서
{**d2, **d1}   # → {'b': 2, 'c': 3, 'a': 1}    ← 'b','c','a' 순서
                # 'b' 는 첫 등장(d2)에서 자리 잡고, 값만 d1 의 2 로 덮임
```

순서를 바꾸면 키 순서와 값 둘 다 바뀜.

### dict 리터럴 vs 키워드 인자 — 표기 헷갈림 정리

```python
{'x': 1}      # ✓ dict 리터럴 (콜론, 키에 따옴표)
{x=1}         # ✗ 문법 에러
dict(x=1)     # ✓ dict 생성자에선 키워드 인자 OK → {'x': 1}
f(x=1)        # ✓ 함수 호출 시 키워드 인자
```

→ dict 리터럴 안은 **콜론(`:`)** 만 사용. 등호(`=`)는 함수 호출/변수 할당용.

### print(컨테이너) 출력은 그대로

```python
print([1, 2, 3])       # → [1, 2, 3]           (대괄호 포함)
print({'a': 1})        # → {'a': 1}            (중괄호 + 따옴표 + 콜론 포함)
print((1, 2))          # → (1, 2)              (소괄호 포함)
```

3단계 규칙 재확인: `print(컨테이너 1개)` → 컨테이너 표기 그대로. `print(값 여러 개)` → 공백 연결.

### 위치별 별표 통합 정리

```
* 의 위치별 의미
├─ 정의 시 매개변수 앞   : 모으기 (튜플)
├─ 호출 시 인자 앞       : 풀기 (위치 인자로)
└─ 컨테이너 리터럴 안     : 풀기 (원소로)

** 의 위치별 의미
├─ 정의 시 매개변수 앞   : 모으기 (dict)
├─ 호출 시 인자 앞       : 풀기 (키워드 인자로)
└─ dict 리터럴 안        : 풀기 (키:값 쌍으로)
```

### 한 줄 요약

> **별표는 `[]`/`()`/`{}` 안에서도 풀린다. dict 합칠 때 자리는 첫 등장, 값은 마지막 쓰기.**

### 오늘 막힌 지점

1. **dict 에 같은 키 두 번 적음** (`{y=2, y=99}`) — dict 키는 유니크. 같은 키 충돌 시 덮어씀.
2. **dict 표기를 `x=1` 로 적음** — dict 는 `'x': 1` (콜론). `=` 은 함수 호출/할당용.
3. **`print(list)` 출력에서 `[]` 누락** — print 는 컨테이너 표기 그대로 출력.
4. **`{**d2, **d1}` 의 키 순서 추측 실패** — 첫 등장한 d2 의 'b','c' 자리가 먼저 잡힌 뒤 d1 의 'a' 가 끝에 추가됨.

## [2026-06-08 18:48] python_syntax — state 누적의 조건 (`return` 던지기 vs `=` 받기)

### 본인 표현 (회상용)

> **"`=` 없으면 저장 안 되서 계속 빙빙 돈다"**

### 핵심: `{**state, **changes}` 는 항상 새 dict 를 만들지 원본을 안 바꾼다

```python
state = {'step': 0, ...}

update_state(state, step=1)   # ← 새 dict 만들어 반환했지만, 안 받음
print(state)                   # → {'step': 0, ...}    ← 그대로!
```

새 dict 를 만들었어도 **변수가 그걸 가리키도록 재할당하지 않으면 사라짐**.

### 비유 — 캐치볼

```
함수: 공(새 dict)을 던짐 (= return)
호출자: 받지 않으면 → 공이 땅에 떨어짐
호출자: state = ... 하면 → 공을 글러브에 받음
```

**둘 다 필요**: 던지는 쪽(`return`) + 받는 쪽(`state =`).

### 빙빙 vs 누적

```python
# 빙빙 (= 없음) — 매 호출이 원본에서 시작
update_state(state, step=1)
update_state(state, step=2)
update_state(state, step=3)
# state 는? → 영원히 step:0

# 누적 (= 있음) — 새 dict 가 변수에 잡힘
state = update_state(state, step=1)
state = update_state(state, step=2)
state = update_state(state, step=3)
# state 는? → step:3
```

### LangGraph 노드의 흔한 버그 자리

```python
# ✗ 흔한 실수
def my_node(state):
    update_state(state, step=state['step']+1)   # 만들기만 하고 안 받음
    return state                                  # 원본 그대로 반환 → 그래프가 멈춤

# ✓ 정상
def my_node(state):
    return {**state, 'step': state['step']+1}   # 새 dict 만들어 즉시 반환
```

"왜 state 가 안 바뀌지?" 의 80%가 이 자리.

### LangGraph 가 이 방식을 택한 이유 (immutable 철학)

매 노드가 **원본 안 바꾸고 새 dict 반환** → 프레임워크가 새 상태 생성 → 다음 노드로 전달.
- 시점별 스냅샷 가능 (디버깅·time-travel)
- 안전한 병렬 처리
- 롤백 가능 — 이전 dict 들 그대로 살아있음

### 한 줄 요약

> **`return` 은 던지기, `=` 은 받기. 둘 다 있어야 누적된다. 한쪽만 있으면 공이 땅에 떨어진다.**
