### 잡힌 개념
- **TypedDict = 내부 통신용 / Pydantic = 외부 경계 검문소** ("국내 자유통행 vs 국경 여권검사")
- **dict라고 자동 key-merge 안 됨** — reducer 없으면 통째로 덮어쓰기. `config: dict`도 덮어쓰기.
- **`with_structured_output(Pydantic)`**: 정규식 추출 아니라 **function calling으로 JSON 강제** → Pydantic 인스턴스 자동 변환
- **ReAct = chatbot ↔ tool 왕복**. chatbot 노드는 하나뿐, state(messages)가 달라져서 매번 다른 응답
- **TypedDict는 진짜 상속 아님** — `class S(TypedDict):`는 메타클래스가 가로채는 스키마 선언. 인스턴스는 그냥 dict.

### 오개념 정정
- "TypedDict가 BaseModel의 하위 종류 아닌가?" → **완전 별개**. typing 표준 vs pydantic 외부 라이브러리.
- Pydantic이 일반적으로 더 많이 쓰이지만 **LangGraph state 자리는 TypedDict가 표준** (dict로 흘러야 reducer 작동)

### 약점 연결
- `dict_immutability_in_merge` — LangGraph 노드 함수 작성법 그대로. `state["x"]=` 금지, `return {"x": ...}` 필수.
- `container_unpacking_merge` — custom reducer 작성 시 `{**left, **right}` 패턴 그대로 활용.

### 실제 에이전트 state 패턴 정리 (RAG / ReAct / Plan-Execute)
- **덮어쓰기 (reducer 없음)**: question, answer, plan, config, grade, current_step — "지금 상태" 류
- **누적 (reducer 있음)**: messages(add_messages), documents(add), past_steps(add) — "여태까지 모은 것" 류
- 핵심: **type이 list/dict라도 의미가 "현재 하나"면 덮어쓰기** (예: `plan: list`는 reducer 없음)
