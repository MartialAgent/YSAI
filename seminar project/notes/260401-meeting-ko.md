# 회의록 — 2026-04-01
**주제:** AgentDAG 평가 아키텍처 논의  
**핵심 문제:** A-Score의 얕은 유사도 평가 한계 극복

---

## 개요

현재 A-Score의 핵심 약점은 코사인 유사도에 의존한다는 점으로, LLM 에이전트를 견고하게 평가하기에는 부족합니다. 이번 회의에서는 평가 아키텍처를 개선하기 위한 7가지 방향을 논의했습니다.

---

## 1. G-Eval — 확률 기반 LLM-as-Judge

### 작동 방식 (3단계)

**1단계: 채점 기준(Rubric) 정의 (Evaluation Prompting)**
- 개발자가 자연어로 평가 기준을 직접 작성
- 정답(Ground Truth) **불필요**
- 예시: *"이 답변의 '전문성'을 1~5점으로 평가해줘. 전문성이란 구체적인 수치를 언급하고, 비즈니스 용어를 정확히 사용하는 것을 말해."*

**2단계: Chain-of-Thought 체크리스트 자동 생성**
- LLM이 점수를 내기 전에 스스로 세부 채점 단계를 작성하도록 유도
- 내부 사고 예시:
  1. 답변에 구체적인 수치(ROI %, 가격 등)가 포함되어 있는지 확인
  2. 'Churn rate', 'LTV' 같은 비즈니스 용어가 쓰였는지 확인
  3. 논리가 비약 없이 연결되는지 확인

**3단계: 폼 작성 + 확률 기반 점수 산출**
- LLM이 체크리스트를 바탕으로 답변을 분석 후 최종 점수 도출
- **핵심:** 단순히 출력된 텍스트("4점")를 쓰는 게 아니라, **각 점수 토큰(1~5)의 Log-probability 가중 평균**을 사용
- 예시: P(4점) = 80%, P(5점) = 20% → 최종 점수 = **4.2점** (더 정밀하고 안정적)

### 활용 플랫폼
- Galileo AI — AI 관측성(Observability) 및 평가 엔지니어링 플랫폼에서 이 방식을 사용하는 것으로 보임

### 우려 사항
> 현재 에이전트 1회 요청 시 API 호출 약 5회 → G-Eval 도입 시 **8~10회**로 증가 가능. 비용 및 지연 트레이드오프 검토 필요.

## 2. A-Score 2.0 — 하이브리드 업그레이드 방향

현재 A-Score의 구성 요소(D, S, C) 각각을 다음과 같이 개선할 수 있습니다:

| 구성 요소 | 현재 방식 | 개선 제안 |
|-----------|----------|----------|
| **S** (유사도) | 코사인 유사도 | SelfCheckGPT 방식의 로컬 **NLI(자연어 추론) 모순 탐지**로 교체 |
| **C** (확신도) | 모델 자체 답변 | **API Logprobs**(토큰 확률 통계값) — 완전 자동 추출 |
| **D** (편차) | 단순 Critic 프롬프팅 | **QAG(질문-답변 쌍 검증)** 또는 **G-Eval 프롬프팅** 도입 |

### 보완 기법

#### Reflexion — 자기 반성 기반 에이전트 평가
- 수치 점수 대신, Critic이 **행동 반성문(Trajectory Feedback)** 을 단기 메모리에 작성
- 워커(Worker) 에이전트가 다음 재시도 시 자신의 반성문을 읽고 개선: *"아까 A 부분을 놓쳤지"*
- 활용: 점수 기반 컷오프(A-Score) 대신 **피드백 루프 자체를 노드 통과 기준**으로 삼는 아키텍처

#### QAG — 질문-답변 쌍 검증 (환각 탐지 끝판왕)
- G-Eval의 주관적 판단보다 훨씬 수학적이고 투명한 방식 (RAGAS에서 애용)
- **작동 방식:**
  1. 이전 노드의 결과물(Input)에서 LLM이 퀴즈 3개를 출제
  2. 현재 노드의 생성물(Output)만 보고 그 퀴즈를 풀 수 있는지 테스트
- **접목:** 퀴즈 정답률 → A-Score의 **D(편차)** 구성 요소로 활용

## 3. 교차 모델 평가 — 다른 LLM을 심사위원으로

**문제 인식 (수업 내용):** GPT는 GPT가 생성한 답변에 더 우호적인 점수를 주는 경향이 있음 (동일 모델 편향).

**제안:** 단일 LLM 심사위원 대신, **여러 다른 LLM 심사위원의 평균 점수**를 평가 기준으로 사용.

> **액션 아이템:**  
> - 관련 논문 탐색 (김재형 교수님 연구 그룹 확인)  
> - 호익님 작년 프로젝트 인사이트 확인 (답변 4개 모델 돌아가며 평가 방식)

## 4. SAC³ — 의미 인식 교차 일관성 검증

**논문:** [arXiv:2311.01740](https://arxiv.org/abs/2311.01740)  
**참고 블로그:** [한국어 요약](https://yoonschallenge.tistory.com/1052)

### 등장 배경
기존 self-consistency가 놓치는 두 가지 오류:
1. **질문 수준 환각 (Question-level hallucination)** — 질문이 모호해서 생기는 불일치
2. **모델 수준 환각 (Model-level hallucination)** — 모델이 일관되게 같은 오답을 반복

### 핵심 방법
- **의미적 변형(Semantic Perturbation):** 같은 의미의 질문을 여러 방식으로 바꾸어 답변 안정성 확인
- **교차 모델 일관성(Cross-model Consistency):** 여러 다른 LLM의 답변을 비교

### 점수 구성 요소

| 점수 | 설명 |
|------|------|
| **SC² (자기 일관성)** | 동일 질문 Q₀에 대한 여러 샘플의 의미 일치도. 낮은 점수 = 높은 일관성 = 사실일 가능성↑ |
| **SAC³-Q (질문 수준)** | 의미적으로 같은 변형 질문 Q₁~Qₖ에서의 답변 일치도. 다른 표현에서도 같은 오답 → 환각 위험↑ |
| **SAC³-M (모델 수준)** | 검증 LM과의 교차 확인. GPT만 "아니오"이고 다른 모델들은 "예" → GPT 환각 가능성↑ |
| **SAC³-QM** | 질문 수준 + 모델 수준 통합 교차 검증 |

**최종 점수 산식:**  
`최종 = SC² + λ × (SAC³-Q + SAC³-M)`  
λ = 검증 LM 신뢰도 가중치 (기본값 1.0, 전문 도메인 모델일 경우 가중치 상향)

## 7. GPT 제안 — 의존성 민감도 점수 (DSS)

**동기:** DAG 구조 에이전트에 특히 중요한 지표.

**정의:**
> "이 노드의 입력을 조금 바꾸면, 하위(downstream) 결과가 얼마나 흔들리는가?"

$$\text{DSS}(\text{node}_i) = \frac{\Delta(\text{Output}_{\text{downstream}})}{\Delta(\text{Input}_i)}$$

- **DSS 높음** = 고위험 상위(upstream) 노드 (작은 입력 변화가 큰 하위 파급 효과 유발)
- A-Score에 **노드 위험 가중치**로 통합 가능

## 부록 — 참고 논문

### "Towards a Science of Scaling Agent Systems"
**arXiv:** [2512.08296](https://arxiv.org/abs/2512.08296)

**5가지 핵심 발견:**

1. **도구-협업 트레이드오프:** 고정된 토큰 예산 내에서, 복잡한 도구 중심 작업은 에이전트 간 소통이 늘어날수록 성능이 저하됨. 협업 비용이 도구 사용에 필요한 인지 예산을 소모하기 때문.

2. **능력 포화 지점:** 단일 에이전트 성능이 이미 **45% 이상**인 작업에서는 에이전트 추가 시 효과 미미하거나 역효과. 충분히 똑똑한 단일 에이전트가 팀보다 나을 수 있음.

3. **구조별 오류 증폭:**
   - 독립형 (소통 없음): **오류 17.2배 증폭**
   - 중앙 집중형 (관리자 감독): **오류 4.4배 증폭**

4. **작업 유형별 최적 구조:**

   | 작업 유형 | 최적 구조 | 성과 |
   |-----------|----------|------|
   | 병렬 가능 (예: 금융 분석) | 중앙 집중형 멀티 에이전트 | +80.8% |
   | 동적 웹 탐색 | 분산형 | 유리 |
   | 순차적 추론 | 단일 에이전트 | 멀티 에이전트 대비 −39~70% |

5. **예측 모델:** 작업 특성만으로 최적 에이전트 구조를 예측하는 모델 개발 — **새로운 작업 환경에서 87% 정확도**

**핵심 시사점:**
> 에이전트를 많이 투입한다고 좋은 것이 아님. **작업의 순차성(Sequentiality)** 과 **도구 사용 밀도(Tool Density)** 를 따져서 설계해야 함.
