# AgentDAG 시스템 기술 명세서

본 문서는 AgentDAG 시스템의 구성 요소, 작동 방식, 그리고 각 파일의 역할을 비기술직 임원도 이해할 수 있는 수준으로 설명합니다.

---

## 0. 용어 사전 (Glossary)

| 용어 | 정의 |
| :--- | :--- |
| **DAG (Directed Acyclic Graph)** | 작업들 사이의 순서와 의존 관계를 정의한 구조도. 작업 A가 끝나야 작업 B를 시작할 수 있다는 관계를 표현하며, 순환(되돌아가기)이 없는 일방향 흐름. |
| **노드 (Node)** | DAG 안에서 하나의 독립적인 작업 단위. 본 시스템에서는 "트렌드 분석", "수익 모델 설계" 등이 각각 하나의 노드에 해당. |
| **에이전트 (Agent)** | 특정 역할을 수행하도록 지시를 받은 AI 모델 인스턴스. 본 시스템에서는 생성 담당(Worker), 검증 담당(Critic), 합의 담당(Consensus)으로 역할이 분리됨. |
| **Worker** | 실제 작업 결과물(답변, 기획서 등)을 생성하는 AI 에이전트. |
| **Critic** | Worker가 생성한 결과물의 논리적 오류, 사실 오류, 근거 부족 등을 검출하는 검증 전용 AI 에이전트. |
| **PFA (Pass / Fail / Ambiguous)** | 본 시스템의 핵심 판정 체계. 각 작업 결과를 통과(Pass), 실패(Fail), 모호함(Ambiguous) 세 가지 상태 중 하나로 분류하는 방법론. |
| **A-Score (Ambiguity Score)** | 특정 작업 결과가 얼마나 모호한지를 0~1 사이의 숫자로 표현한 지표. 0에 가까우면 신뢰할 수 있고, 0.7을 넘으면 사람의 확인이 필요한 상태로 판정. |
| **S (Similarity, 유사도)** | AI에게 같은 질문을 두 번 했을 때 두 답변이 얼마나 일치하는지를 측정한 수치 (0~1). 높을수록 AI의 답변이 일관적임을 의미. |
| **C (Confidence, 확신도)** | AI가 자기 답변이 얼마나 정확하다고 스스로 평가한 수치 (0~1). |
| **D (Deviation, 편차)** | Critic 에이전트가 Worker의 답변에서 발견한 오류 및 논리적 결함의 심각도 (0~1). 0이면 결함 없음, 1이면 전면적 결함. |
| **I-Factor (Impact Factor, 영향도 지수)** | 특정 노드의 결과가 수정되었을 때, 그 변경이 후속 노드에 미치는 영향의 크기를 수치화한 지표. 이 값이 임계치(0.4)를 넘는 후속 노드만 다시 실행. |
| **R (Reliance, 의존도)** | 후속 노드가 이전 노드의 결과 데이터를 얼마나 직접적으로 활용하는지를 나타내는 가중치 (0~1). 0.9이면 거의 전적으로 의존, 0.2이면 참고 수준. |
| **Partial Back-propagation (부분 소급 수정)** | 오류가 수정되었을 때, 전체 작업을 처음부터 다시 수행하지 않고 영향을 받는 후속 작업만 선별적으로 다시 수행하는 최적화 기법. |
| **Human-in-the-loop (HITL)** | AI가 스스로 판단하지 못하는 모호한 상황(Ambiguous)에서 시스템 실행을 중단하고, 사람에게 확인을 요청한 뒤 그 결과를 반영하여 재개하는 운영 방식. |
| **API (Application Programming Interface)** | 외부 AI 서비스(OpenAI 등)에 요청을 보내고 응답을 받기 위한 통신 규약. 본 시스템은 OpenAI의 ChatGPT API를 호출하여 텍스트를 생성하고 검증함. |
| **토큰 (Token)** | AI 모델이 텍스트를 처리할 때 사용하는 최소 단위. API 사용 비용은 소모된 토큰 수에 비례하여 과금됨. 대략 한글 1글자 ≒ 2~3 토큰. |
| **임계값 (Threshold)** | 판정 기준이 되는 경계 수치. 예를 들어 A-Score의 임계값이 0.7이면, 0.7 이하는 통과(Pass), 0.7 초과는 모호함(Ambiguous)으로 분류. |
| **JSON** | 데이터를 저장하고 전달하기 위한 텍스트 기반 형식. 본 시스템에서 작업 진행 상태와 결과를 기록하는 데 사용됨. |

## 1. 시스템이 하는 일 (System Purpose)

AgentDAG는 여러 개의 AI 에이전트가 순서대로 작업을 수행하는 과정에서 다음 세 가지를 자동으로 관리합니다.

1. **결과물 품질 판정**: 각 작업 결과를 수치 기반으로 Pass(통과) / Fail(실패) / Ambiguous(모호함)로 분류합니다.
2. **비용 최적화 복구**: 중간 결과가 수정되었을 때 전체를 다시 수행하지 않고, 영향을 받는 후속 작업만 다시 수행합니다.
3. **사람 개입 요청**: AI가 스스로 판단할 수 없는 모호한 상황에서만 사람에게 확인을 요청합니다.

## 3. 개발된 파일별 상세 설명 (Developed Files)

### 3.1 `main.py` — 시스템 실행 시작 파일

**역할**: 전체 시스템을 가동시키는 진입점입니다.

**수행하는 작업**:
- 테스트용 DAG 그래프를 정의합니다. 현재는 3개의 노드로 구성되어 있습니다.
  - 노드 A: 2026년 자율주행 시장 트렌드 3가지 분석
  - 노드 B: 노드 A의 결과를 기반으로 구독형 수익 모델 설계
  - 노드 C: 노드 B의 결과를 기반으로 마케팅 이메일 제목 5개 생성
- 그래프 실행을 시작하고, Ambiguous(모호함) 상태가 발생하면 터미널에서 사용자 입력을 받아 수정된 데이터로 재개합니다.

---

### 3.2 `src/config.py` — 판정 기준값 및 AI 모델 설정

**역할**: 시스템의 모든 조정 가능한 수치를 한 곳에서 관리합니다.

**포함된 설정값**:

| 설정 항목 | 현재 값 | 의미 |
| :--- | :--- | :--- |
| ALPHA ($\alpha$) | 0.5 | A-Score 계산 시 유사도(S) 항에 부여하는 가중치 |
| BETA ($\beta$) | 0.5 | A-Score 계산 시 편차(D) 항에 부여하는 가중치 |
| A_SCORE_THRESHOLD | 0.7 | 이 값을 초과하면 '모호함(Ambiguous)' 판정 |
| I_FACTOR_THRESHOLD | 0.4 | 이 값을 초과하는 후속 노드만 재실행 대상으로 선정 |
| MAX_FAIL_RETRIES | 2 | 실패(Fail) 시 자동 재시도 최대 횟수. 초과 시 Ambiguous로 격상 |
| WORKER_MODEL | gpt-4o-mini | 결과물 생성에 사용하는 AI 모델 (비용 효율적) |
| CRITIC_MODEL | gpt-4o | 결과물 검증에 사용하는 AI 모델 (정밀도 우선) |
| EMBEDDING_MODEL | text-embedding-3-small | 두 텍스트 간 유사도 측정에 사용하는 AI 모델 |

---

### 3.3 `src/models.py` — 데이터 구조 정의

**역할**: 시스템 내에서 사용되는 모든 데이터의 형태(구조)를 사전에 정의합니다.

**정의된 데이터 구조**:

1. **NodeStatus (노드 상태)**: 각 노드가 가질 수 있는 5가지 상태값.
   - `IDLE`: 아직 실행되지 않음
   - `RUNNING`: 현재 실행 중
   - `PASS`: 품질 기준 통과
   - `FAIL`: 명백한 오류 감지
   - `AMBIGUOUS`: 판단 불가, 사람의 확인 필요

2. **EvaluationResult (판정 결과)**: 하나의 노드에 대해 판정 과정에서 산출된 수치 집합.
   - 유사도(S), 확신도(C), 편차(D), 최종 모호함 지수(A-Score), 그리고 Critic의 서면 피드백을 포함.

3. **NodeState (노드 상태 전체)**: 하나의 작업 노드에 대한 모든 정보를 담는 단위.
   - 작업 설명, 입력 데이터, 출력 데이터, 현재 상태, 판정 결과, 재시도 횟수, 의존하는 상위 노드 목록을 포함.

4. **GraphState (그래프 전체 상태)**: 시스템 내 모든 노드를 하나의 집합으로 관리하는 최상위 구조.

---

### 3.4 `src/evaluator.py` — 품질 판정 로직

**역할**: 각 노드의 AI 생성 결과물이 신뢰할 수 있는지를 수치로 판별하는 핵심 엔진입니다.

**작동 절차** (한 노드당 총 4회의 AI 호출):

| 단계 | 호출 대상 | 수행 내용 | 산출 지표 |
| :--- | :--- | :--- | :--- |
| **1단계: 1차 생성** | Worker (gpt-4o-mini) | 주어진 작업에 대한 답변 생성 | 최종 결과물 |
| **2단계: 2차 생성** | Worker (gpt-4o-mini) | 동일한 작업에 대해 한 번 더 답변 생성 | 유사도(S) 비교용 |
| **3단계: 자기 평가** | Worker (gpt-4o-mini) | 1차 답변이 얼마나 정확한지 스스로 평가 | 확신도(C) |
| **4단계: 외부 검증** | Critic (gpt-4o) | 1차 답변의 논리적 결함 및 사실 오류 검출 | 편차(D), 서면 피드백 |

**판정 수식**:
$$A = \alpha \times (1 - S) + \beta \times D \times (1 - C)$$

**판정 기준**:
- 편차(D)가 0.7 이상이면 → **FAIL** (명백한 오류)
- A-Score가 0.7을 초과하면 → **AMBIGUOUS** (사람 확인 필요)
- 그 외 → **PASS** (통과)

### 3.5 `src/orchestrator.py` — 작업 순서 관리 및 복구 로직

**역할**: 노드들의 실행 순서를 결정하고, 판정 결과에 따라 진행/중단/재실행을 관리하는 중앙 제어기입니다.

**핵심 기능 4가지**:

1. **실행 순서 결정 (Topological Sort)**
   - 노드 간의 선후 관계(의존성)를 분석하여 올바른 실행 순서를 자동으로 산출합니다.
   - 예: 노드 B가 노드 A에 의존하면, 반드시 A → B → C 순서로 실행됩니다.

2. **자동 재시도 및 격상 (Retry & Escalation)**
   - Fail(실패) 판정 시, Critic의 피드백을 반영하여 자동으로 최대 2회까지 재시도합니다.
   - 2회 재시도 후에도 실패하면, 해당 노드를 Ambiguous로 격상시켜 사람의 확인을 요청합니다.

3. **영향도 기반 부분 재실행 (Partial Back-propagation)**
   - 사람이 특정 노드의 결과를 수정하면, 후속 노드들 중 영향을 받는 노드만 선별하여 다시 실행합니다.
   - 선별 공식: $I_{total} = \sum (R / (d + 1))$
     - R: 후속 노드가 수정된 데이터에 의존하는 정도
     - d: 수정된 노드로부터의 거리
   - 이 값이 0.4를 초과하는 노드만 재실행 대상으로 지정됩니다.

4. **상태 자동 저장 및 복구 (Persistence)**
   - 매 노드 완료 시마다 전체 진행 상태를 `state.json` 파일에 자동 저장합니다.
   - 시스템이 예기치 않게 종료되더라도, 마지막으로 성공한 지점부터 이어서 실행할 수 있습니다.

## 4. 자동 생성되는 파일 (Generated Artifacts)

### 4.1 `state.json` — 실행 결과 및 진행 상태 기록

**생성 시점**: `main.py`를 실행하면 자동으로 생성 및 갱신됩니다.

**기록되는 정보** (노드별):

| 필드명 | 내용 |
| :--- | :--- |
| `node_id` | 노드의 고유 식별자 (예: "A_Trend") |
| `task_description` | 해당 노드에 부여된 작업 지시문 |
| `input_data` | 이전 노드로부터 전달받은 데이터 |
| `output_data` | AI가 생성한 최종 결과물 (텍스트 전문) |
| `status` | 현재 판정 상태 (PASS, FAIL, AMBIGUOUS 중 하나) |
| `evaluation` | 유사도(S), 확신도(C), 편차(D), A-Score, Critic 피드백 |
| `retry_count` | 자동 재시도가 수행된 횟수 |
| `depends_on` | 의존하는 이전 노드의 ID와 의존도(R) 수치 |

**활용 목적**:
- 실험 결과의 정량적 분석 및 보고서 작성의 원천 데이터.
- 시스템 중단 시 복구(Resume) 지점 결정의 근거 자료.
- 향후 임계값 조정(튜닝) 시 과거 판정 이력 참조 자료.

## 5. 비용 구조 (Cost Structure)

하나의 노드를 판정하기 위해 발생하는 AI API 호출 내역:

| 호출 | 사용 모델 | 목적 | 예상 비용 |
| :--- | :--- | :--- | :--- |
| Worker 1차 생성 | gpt-4o-mini | 결과물 생성 | 낮음 |
| Worker 2차 생성 | gpt-4o-mini | 일관성(S) 측정용 | 낮음 |
| Worker 자기 평가 | gpt-4o-mini | 확신도(C) 산출 | 낮음 |
| Critic 검증 | gpt-4o | 편차(D) 및 피드백 | 중간 |
| Embedding 유사도 | text-embedding-3-small | 두 답변 간 S 계산 | 매우 낮음 |

**노드당 총 5회 호출**. 단, 이 판정 비용은 오류 발생 시 전체 워크플로우를 처음부터 재실행하는 비용을 방지하기 위한 **사전 투자**에 해당합니다.
